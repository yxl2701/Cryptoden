# Cryptoden - CTF Cryptography AI Encryption/Decryption Tool

Cryptoden is a Python toolbox for CTF cryptography problems, providing both graphical user interface (GUI) and command-line interface (CLI) usage.

The project supports common classical ciphers, encoding/decoding, hashing, symmetric cryptography, asymmetric cryptography, and RSA/ECC/LCG attack modules, with built-in AI analysis and dialogue capabilities.

The current project uses the `algorithms/` directory as a unified algorithm entry point. The program dynamically scans algorithm files and automatically loads capabilities based on functions like `encrypt()`, `decrypt()`, and `decrypt_all()`. GUI parameter panels and CLI parameters are also automatically identified based on function signatures, eliminating the need for each algorithm to manually maintain `PARAMS`.

The project uses **AI-assisted development**, with some code and features generated by AI and verified through existing automated testing. For complex attack modules, users are still advised to manually verify results based on the problem environment.

The project is open-sourced under the MIT License, and users are welcome to perform secondary development, submit Issues, or contribute improvements.

## Project Overview and Review

Project positioning:

- Focused on CTF cryptography and common encryption/decryption tasks, providing GUI, CLI, AI assistance, and attack module entry points.
- Regular algorithms use a dynamic loading architecture, reducing invasive modifications to the main interface and CLI when adding new algorithms.
- Complex functions like RSA/ECC/LCG/AES retain dedicated interfaces to avoid cramming numerous parameters into the generic panel.

Current review:

- The algorithm directory and loading method have been unified, making the cost of extending regular algorithms relatively low.
- The main interface algorithm tabs have been changed to configuration-driven, with only `RSA` and `AES` displayed by default; other tabs are enabled by users themselves.
- The AI workstation has been split into regular dialogue and restricted Agent mode. The Agent restricts file, command, network, and sensitive configuration access through local policies and MCP tool layer.
- Windows clipboard rich text MIME errors have been avoided through plain text copy processing.
- SageMath support includes local, WSL, and SageCell online execution, but complex attacks are still affected by network, environment, and parameter scale.
- Project paths are resolved based on the source root directory, avoiding reliance on absolute directories on the developer's machine; except for external environment configurations like SageMath executables, WSL paths, and API addresses.
- Default AI configuration does not include real API Keys or chat history; logs, caches, build artifacts, and local decryption records should still be cleaned before release.

## Portability Notes

- When starting `python main.py`, `python cli.py ...`, or packaged programs from any directory, internal resources will be located based on the source root directory or packaged resource directory.
- `config.json`, `algorithm_tabs_config.json`, `sagemath_config.json`, and `config/ai_config.json` are saved using relative project positions, so there's no need to modify internal paths after moving the entire project directory.
- `logs/`, `__pycache__/`, `.pytest_cache/`, `build/`, `dist/`, and `config/decrypt_records.json` are runtime or build artifacts that can be cleaned with `python cleanup.py --apply`.
- `local_path` and `wsl_path` in `sagemath_config.json`, and `api_base` in AI configuration describe external environments and should not be written as project-internal paths; they can be reconfigured in GUI settings after migrating to a new machine.
- Before publishing or sharing source code, do not distribute personal API Keys, chat history, logs, or decryption records as project configurations.

## GUI Features

The main window displays the basic panel and common algorithm tabs by default:

- `Encrypt/Decrypt`: Select algorithm, input text, set parameters, view results.
- `AI Workstation`: Configure and call AI interface for ciphertext analysis, dialogue, or running restricted Agent.
- `RSA`: RSA encryption, decryption, and attack assistance.
- `AES`: Dedicated symmetric cryptography interface, defaulting to AES.

Other symmetric/asymmetric algorithm tabs are hidden by default and can be enabled in `Settings -> Preferences -> Algorithm Tabs`. The list of available algorithm tabs is located in `algorithm_tabs_config.json`, and user selection status is saved to `enabled_algorithm_tabs` in `config.json`.

Menu functions:

- `File`: Open file, save result, refresh algorithm list, restart, exit.
- `Edit`: Undo, redo, paste, clear all.
- `Encrypt/Decrypt`: Lists classical ciphers, encoding/decoding, hashing, and other general algorithms by category; symmetric/asymmetric algorithms are not repeated here.
- `Symmetric Cryptography`: Opens dedicated interfaces for AES/DES/3DES/Blowfish/RC4, etc.
- `Asymmetric Cryptography`: Opens RSA, ECC, LCG dedicated interfaces and attack assistance.
- `Convert`: Case conversion, remove whitespace, remove newlines, base conversion, reverse text.
- `Settings`: Preferences, AI configuration, auto-copy results, auto-swap, decryption records.
- `Help`: Shortcuts, searchable algorithm list, about.

## One-Click Decryption and Recursive Decryption

- One-click decryption attempts directly callable `decrypt()` and automatically calls `decrypt_all()` brute-force candidates, suitable for lightweight scenarios like encoding and classical ciphers.
- One-click decryption expands small key space brute-force results from `decrypt_all()` individually, instead of mixing entire brute-force text blocks in one result.
- One-click decryption results are sorted by keyword hits, flag/ctf format, English readable words, readable character ratio, and result length score, with high-confidence and more readable results displayed first.
- For algorithms returning parameterized brute-force results like Caesar, Affine, and Rail Fence, CLI/GUI will display parameters, e.g., `caesar Caesar cipher - brute-force results / shift=3`.
- `try-all --json` returns `name`, `result`, `is_brute`, `params`, and `score`, making it convenient for scripts to read brute-force parameters and sort scores.
- GUI and CLI one-click decryption share the same `CryptoLoader.try_decrypt_all()` core logic to avoid inconsistent results.
- Recursive decryption attempts multi-layer decryption chains and uses matching rules and result characteristics to filter results, suitable for multi-layer nested encoding/classical ciphers.
- GUI recursive decryption shares the same core logic as CLI, with consistent results, paths, and speed.
- Recursive decryption automatically identifies result characteristics and prioritizes more likely next-step decoding chains. Supported characteristics include:
  - Encoding layers: `Hex candidate`, `Base32 candidate`, `Base58 candidate`, `Base64 candidate`, `URL encoding candidate`
  - Classical layers: `Morse candidate`, `ROT candidate`, `ROT47 candidate`
  - General layers: `flag format`, `braced text`, `alphanumeric candidate`, `highly readable text`, `keywords`
- Base64/Base32 candidates actually verify whether decoding results are printable UTF-8 before marking, avoiding false positives.
- ROT/classical layer candidates have built-in priority sorting; ROT and ROT47 explore independently when they don't conflict.
- For small key space algorithms implementing `decrypt_all()`, one-click and recursive decryption automatically use brute-force results, e.g., Caesar offset brute-force.
- Recursive decryption writes brute-force parameters to decryption paths, e.g., `caesar Caesar cipher(shift=3)`, for easy reproduction.
- Short texts consisting of letters and spaces are used as ROT/Caesar candidates in recursive decryption, e.g., `khoor zruog` can hit `hello world` via `-p hello`.
- Recursive decryption uses multi-level caching (characteristic cache, decryption candidate cache, algorithm lowercase index) to reduce redundant computation.
- Symmetric cryptography (AES/DES/Blowfish, etc.) serves as the last resort layer to prevent them from grabbing budget when there are no characteristics.
- Identical ROT algorithms automatically prune after 3 or more consecutive repetitions to avoid infinite loops.
- Recursive decryption parameters are saved in `recursive_decrypt` of `config.json`, including maximum depth, total attempts, candidates retained per layer, GUI maximum display count, etc.
- Hashing, complex modern cryptography, and algorithms requiring manual parameters/attack context are generally not suitable for automatic one-click decryption.

## Adding New Algorithms

Add new `.py` files under `algorithms/<category>/` and implement functions as needed:

```python
ALGORITHM_NAME = "Example Algorithm"
ALGORITHM_DESC = "Algorithm description"

def encrypt(plaintext, key="default"):
    return plaintext

def decrypt(ciphertext, key="default"):
    return ciphertext

def decrypt_all(ciphertext):
    return []
```

Notes:

- `ALGORITHM_NAME` and `ALGORITHM_DESC` are optional; if not specified, the filename is used by default.
- The first text parameter can be named `plaintext`, `ciphertext`, `cryptotext`, `text`, `input_text`, `message`, `msg`, `input`, or `data`.
- Parameters other than the text parameter will be processed by GUI/CLI as algorithm parameters.
- After adding new files, you can click `File -> Refresh Algorithm List` in the GUI, or restart the program.
- Small key space algorithms are recommended to implement `decrypt_all(ciphertext)`. One-click decryption will expand returned content individually, recursive decryption will read these brute-force candidates, and write parameters to decryption paths.
- The recommended brute-force output format is one candidate per line, e.g., `offset 3: hello world`, `shift 3: hello world`, or `a=1,b=3: hello world`. Text before the colon will be displayed as parameter description.

## AI Workstation

The AI workstation contains two modes:

- `Dialogue Mode`: Uses the configured model for ciphertext analysis, problem explanation, and general dialogue.
- `Agent Mode`: Assists in analyzing current project code under restricted permission policies.

AI configuration file is located at `config/ai_config.json`, and can also be modified in the GUI via `Settings -> AI Configuration...`. Do not submit real API Keys to public repositories.

The model list in AI configuration is fetched in real-time via the current API address; when fetching fails, manual input capability is retained without relying on a fixed model list.

Agent permission configuration is located at `agent/agent_config.json`. The current default permission mode is `command`, and the Agent accesses capabilities through the MCP tool layer:

- Built-in MCP tools include `list_files`, `read_file`, `search_text`, `write_file`, `web_search`, and `run_command`.
- `read_file`, `list_files`, `search_text` can only access non-sensitive paths within the workspace.
- `write_file`, `web_search`, and external MCP tools require user confirmation in the GUI by default.
- `run_command` is still controlled by command whitelist, dangerous command interception, and user confirmation.
- Users can add stdio MCP servers in `mcp_servers` and add `service.tool` to `enabled_mcp_tools` for use.
- The Agent still cannot read full sensitive configuration, nor modify permission configuration, AI Keys, or files outside the project.

For custom MCP configuration examples, see `agent/README.md`.

## SageMath Configuration

RSA attacks marked with `(sagemath)` execute SageMath code through `core/sage_executor.py`. You can configure the execution method in the GUI via `Settings -> Preferences -> SageMath`.

Three methods are supported:

- `Local Installation`: Fill in the local `sage` or `sage.exe` path.
- `WSL`: Fill in the WSL distribution and SageMath path within WSL, e.g., `/usr/bin/sage`.
- `Web SageCell`: Uses `https://sagecell.sagemath.org/` for online execution, no local SageMath installation required.

Testing the connection may cause a brief pause (in testing, web connection typically takes within 10 seconds), which is normal. A test result prompt will appear after the test completes.

Web online execution flow:

- Prioritizes requesting the SageCell `/service` endpoint.
- If the endpoint is unavailable (e.g., returns `HTTP 520`), automatically uses Playwright headless Chromium to open the SageCell page for execution.
- If Playwright is unavailable, uses Selenium Manager headless Chrome as a fallback.

First-time use of Playwright in a new environment may require installing browser kernels:

```bash
python -m playwright install chromium
```

Currently verified online execution can run normal Sage code, finite fields, polynomials, matrices, RSA small exponent/broadcast/shared prime factor, p high-bit leakage, Wiener-type small private key attacks, and other modules. Online environments are affected by network and SageCell service status; complex large integer attacks may still timeout.

## Cleaning Generated Files

The project provides a safe cleanup script for deleting caches, logs, build artifacts, and local decryption records, etc. Cleanup rules are located in `cleanup_config.json`.

By default, only previews without deleting any files:

```bash
python cleanup.py
```

Confirm the list is correct before executing deletion:

```bash
python cleanup.py --apply
```

The script only deletes targets listed in the configuration and is not used to delete source code, documentation, or test files.
